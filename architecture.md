# Архитектура масштабируемого решения

## Задача
500 товаров × 20 запросов = **10 000 проверок в день**
Алерты в Telegram при падении позиции ниже 50.

---

## Технологический стек

### Оркестрация задач: **Celery + Redis**
- Celery — распределённая очередь задач
- Redis — брокер сообщений + кэш результатов
- **Почему:** Celery нативно поддерживает rate limiting, retry,
  приоритеты задач. Redis быстрее RabbitMQ для нашего объёма.

### База данных: **PostgreSQL + TimescaleDB**
- TimescaleDB — расширение для временных рядов
- **Почему:** Данные позиций — это time-series. TimescaleDB даёт
  автоматическое партиционирование по времени, быстрые агрегации.

### Парсинг: **Selenium + undetected-chromedriver**
- Пул из 5-10 воркеров Celery
- Каждый воркер — отдельный Chrome процесс
- **Почему:** Ozon требует JS-рендеринга, requests/httpx не подойдут.

### Мониторинг позиций: **Python сервис + asyncio**
- Асинхронная проверка новых результатов в PostgreSQL
- Сравнение с предыдущими значениями

### Алерты: **python-telegram-bot**
- Webhook или polling
- Шаблоны сообщений с markdown форматированием

### Планировщик: **Celery Beat**
- Cron-расписание: 10 000 задач / 24 часа ≈ 7 задач/минуту
- Равномерное распределение нагрузки
